Description
HIGH CONCEPT:
A multiplayer action shooter set in space where players work together to exterminate pests and compete to collect the most loot without dying.

FEATURES:
3D MOVEMENT AROUND SPHERICAL PLANETS
Planets with local gravity that allow players and other entities to move around on their surface.

RANDOMLY GENERATED LEVELS & EVENTS
Planets are individually designed in chunks to populate the world procedurally.

COMPETITIVE RISK & REWARD STYLE MULTIPLAYER
Players compete to collect the most loot in an increasingly dangerous environment, where if you die, you lose everything.

We had all kinds of random playtests this milestone as we approached the submission deadline. Usually, we were doing general playtests to search for bugs that needed fixing. PLAYTEST NAME:
Beta Feedback Playtest Reports

TOTAL PLAYTEST TIME:
25 minutes

PLAYTESTING GOAL:
The purpose of this playtest was to test the stability of our game on a low-end machine.

PLAYTESTERS:
Ian Aemmer, Michael Van Zant, Glen Aro

NOTES:
Nearing the completion of our game we felt the need to make sure our game was working as we expected it to be. We tested the multiplayer capabilities of our game with several members of Paper Mill and Space Tube. This playtest revealed a major bug with the sprite system in our game. As the game was being played and the players were all firing weapons and bugs were dying the game started to experience major frame rate drops. When this happened any and all sprites on the HUD and the jump menu started to flicker and randomly change to difference sprites found in our game. We restarted the game with fewer pest enemies to see if this was a performance issue. Unfortunately, this problem kept occurring, although we were able to draw some correlation to performance and the sprite flickering occurrence.

FIX:
We optimized the AI by switching from Fernando's state machines to Auston's implementation of the AI. While this helped a little on the performance it did not completely fix the problem. We rebuilt the sprites in the game and got rid of all unused assets to see if this would help, but it also didn't make a difference. After a lot of searching, we came to find out that this was a Unity-specific bug that was only fixable by upgrading from Unity 5.3.x to Unity 5.4.x. After doing so we rebuilt the game and the problem was solved.

PLAYTEST NAME:
Team Playtest Report

TOTAL PLAYTEST TIME:
15 minutes

PLAYTESTING GOAL:
The purpose of this playtest was to test the HUD to see if HUD feedback made sense and made sense.

PLAYTESTERS:
Enrique Rodriguez and Riley Bayless.

NOTES:
While the point of this playtest was to test the HUD and how it looked and worked, Enrique instead found a major bug involving opening the pause menu when the jump menu was also open. This game would cause the entire game to seize up and crash. Enrique and Riley tested this multiple times to see what exact sequence would cause the crash. Upon playtesting and getting reliable crashes we found that the game would crash when the jump menu was open and "B" was pressed.

FIX:
This problem took a lot of time to completely figure out, but it also really optimized the jump and pause menus. The problem ended up being how the input system registering both the inputs at the same time for the jump and pause menu which resulted in an illegal operation while the game was paused. I was able to change the input system to not register any jump menu input when the game was paused and as a safeguard I added the ability to close all active jump menus before the pause menu is opened.
